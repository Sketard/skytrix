## Challenges & Growth Areas

1. **MSG_WIN → DUEL_END gap (Story 3.4 H1)** — The most fundamental duel ending path (natural game end via LP=0 or deck-out) was not covered until the last story. `broadcastMessage()` didn't detect MSG_WIN from the worker. Code review AI caught it, but in production, natural game endings would have caused duels to freeze.

   **Root cause analysis:**
   - `epics-pvp.md` (Story 1.2) listed MSG_WIN as a Server→Client game message in ws-protocol.ts
   - Implementation Readiness Report flagged T2: "OCGCore produces MSG_WIN (ID 5), not DUEL_END" — but classified it as LOW severity
   - Each story handled its own forced-end path (surrender, timeout, inactivity, disconnect) but the natural path was never explicitly assigned to an AC
   - **Systemic gap:** spec assumed the MSG_WIN → DUEL_END mapping was obvious; no story owned it explicitly

2. **Worker lifecycle during rematch (Story 3.4)** — Worker `exit` handler called `cleanupDuelSession()` when `terminate()` was called during rematch. Required `removeAllListeners()` before terminate. Non-obvious edge case in worker lifecycle management.

3. **SCSS budget exceeded (Story 3.4)** — `angular.json` `anyComponentStyle` budget increased from 6kB to 10kB. Pre-existing deck-builder was already at 6.12kB. Result overlay styles pushed it over. Risk for Epic 4 with 6+ animation types.

4. **Absolute vs. relative player indices (Story 3.2)** — Server sends absolute player indices (0/1), client BOARD_STATE sanitization makes turnPlayer relative (0=self, 1=opponent). Required `absoluteTurnPlayer` computed in PvpBoardContainerComponent to bridge the gap for timer badge. This mismatch affects Epic 4 animations (TODO already in message-filter.ts:150).
