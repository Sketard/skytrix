## Successes & Strengths

1. **Architecture "Immediate State + Overlay"** — Story 4.2's pragmatic decision to apply BOARD_STATE immediately and run animations as CSS overlays preserved 3 epics of stable state handling. No structural change to existing BOARD_STATE processing. Visual result equivalent to deferred model with half the complexity.

2. **Chain badges / animation queue clean separation** — Story 4.1 introduced `activeChainLinks` as persistent visual state (badges that appear, stay, disappear). Story 4.2 introduced `animationQueue` as transient effects (play once, consume). Two independent signals, zero conflict. MSG_CHAINING triggers both simultaneously without interference.

3. **DRY KISS adherence** — Intentionally skipped complex animations (MSG_ATTACK line animation, MSG_BATTLE, MSG_DRAW, MSG_SWAP). Story 4.1 used `zoneId: null` for non-field activations instead of over-engineering floating badges. Axel directive from Epic 3 retro respected throughout.

4. **Previous Story Intelligence — 0 recurring bugs** — Anti-patterns documented in every story (`[class]` binding, mutable signal updates, missing `untracked()`, inline z-index/colors). Zero recurrence across both stories. Practice continues to deliver measurable value since Epic 1.

5. **Code review AI — critical safety net** — Caught 1 CRITICAL (LP reduced-motion), 2 HIGH (LP race condition, spec divergence), and 17 M/L findings. All fixed before completion. The review system remains the project's primary quality gate.

6. **Signal-based architecture consistency** — Both stories maintained signal-based state patterns, standalone components, Angular 19 control flow, OnPush change detection. No deviation from established conventions across all 4 epics.

7. **Zero build errors, SCSS budget not exceeded** — Despite ~90 lines of animation CSS added across 2 components, the 10kB `anyComponentStyle` budget (raised in Epic 3) was not breached.

## Challenges & Growth Areas

1. **prefers-reduced-motion CRITICAL regression (Story 4.2 C1)** — LP counter `requestAnimationFrame` used hardcoded `duration = 500` with no snap under `prefers-reduced-motion`. WCAG accessibility violation. The existing checklist covered CSS `@keyframes` and `animation` properties but NOT JavaScript-driven animation patterns (rAF, setTimeout-based interpolation).

   **Root cause analysis:**
   - Epic 2 introduced the `prefers-reduced-motion` checklist targeting CSS animations
   - Epic 3 achieved 0 findings — all animations were CSS-based
   - Epic 4 introduced a NEW animation pattern: `requestAnimationFrame` for LP counting
   - The checklist was CSS-specific, not animation-pattern-agnostic
   - **Systemic gap:** checklist scope didn't evolve with the introduction of new animation patterns

2. **Findings per story uptick (7 → 10)** — 25% of findings (5/20) were hardcoded values instead of design tokens. Epic 4 introduced significantly more design tokens (8+ animation tokens) than previous epics (2-3). More tokens = more opportunity for error. Specific findings: C1 hardcoded duration, M1 hardcoded rgba, M2 LP flash hardcoded colors, M4 wrong token name, L1 wrong token name.

3. **LP tracking race condition (Story 4.2 H1)** — `trackedLp` was reset by the `duelState` effect before pending MSG_DAMAGE was processed, causing double-subtraction and LP bounce. Fix: guard `trackedLp` reset behind `!_isAnimating()`, explicit sync at end of `processAnimationQueue()`. Symptom of increased async complexity with animation orchestration.

4. **Temporal complexity leap** — Epic 1-3 were primarily state management (signals, computed, effects). Epic 4 introduced temporal choreography (animation queues, setTimeout durations, rAF interpolation, prompt drain coordination, queue collapse). This is a fundamentally different category of complexity that explains the findings uptick.

## Key Insights & Learnings

1. The `prefers-reduced-motion` checklist must cover ALL animation patterns (CSS, rAF, setTimeout), not just CSS — new patterns require checklist expansion
2. More design tokens = more error surface — token compliance needs explicit review verification
3. The code review AI is the critical safety net — findings trended up but ALL were caught and fixed
4. "Immediate State + Overlay" is a reusable architectural pattern for animation systems that need to coexist with reactive state management
5. Previous Story Intelligence + declining then stabilizing findings trend = process maturity is real but has limits when complexity class changes
6. Temporal choreography (animation queues, timing, drain coordination) is a fundamentally different complexity class from state management — expect higher initial findings rate
