## Successes & Strengths

1. **Exemplary incremental architecture** — 7.1 foundation (service + registry) → 7.2 integration (MSG_MOVE travel) → 7.3 buffering (chain replay) → 7.4 extension (MSG_DRAW + shuffle) → 7.5 enrichment (XYZ materials). Zero destructive refactoring between stories.

2. **Web Animations API — right technology choice** — Eliminated CSS/JS timing sync issues from Epic 6 (H3 bug). All travel animations driven purely by JS, avoiding parallel CSS keyframe synchronization problems.

3. **Buffer & replay architecture** — Elegant solution for chain resolution visibility. Events buffered during overlay, replayed in two beats (zone travels then LP) after overlay hides. `scheduleTimeout` + `.then()` pattern preserves cancellability without async/await.

4. **Best-ever findings ratio (5.2/story, 0 CRITICAL)** — Down from 13 (Epic 1) → 8 → 7 → 10 → **5.2**. Previous Story Intelligence sections drove zero recurring bugs across all 5 stories.

5. **Clean layer separation maintained** — Orchestrator (timing) ↔ CardTravelService (visual/floating elements) ↔ Board container (DOM/zone registry). Fire-and-forget `travel()` pattern held across 4 stories without modification.

6. **Fire-and-forget + sync duration return pattern** — `processMoveEvent()` calls `travel()` without awaiting, returns base duration (400ms) for queue setTimeout. Proven robust across MSG_MOVE, MSG_DRAW, and overlay detach events. Reusable for any future animation type.

## Challenges & Growth Areas

1. **Reduced-motion guard not systematic on new methods** — `_reducedMotion` early-return was missing in `replayBufferedEvents()` (7.3 H-fix) and `processDrawEvent()` (7.4 H-fix). The guard exists at CardTravelService level but each orchestrator method that manages timing needs its own guard. This is the 3rd consecutive epic with reduced-motion findings.

   **Root cause:** The checklist was expanded after Epic 4 (CSS + rAF + setTimeout) but the rule wasn't formalized as "every method returning duration > 0 needs the guard." New methods slip through because developers assume the service-level guard is sufficient.

2. **Unverified OCGCore data assumptions** — Token dissolution implemented in 7.2 based on assumption `cardCode === 0` means token, but it's ambiguous with sanitized opponent cards. Removed in code review. `LOCATION.OVERLAY` (0x80) was missing from protocol constants, discovered only during 7.5 implementation — the value was already sent by the server but silently unrecognized client-side.

3. **CSS keyframes vs composed transforms** — Story 7.4 shuffle animation was broken because pseudo-elements had existing base transforms that the `@keyframes` overwrote instead of composing. Required splitting into separate keyframes per pseudo-element.

4. **Low-priority debt never addressed** — Card inspector placeholder and Docker integration test carried since Epic 1 (6 epics). Directive from Axel: resolve all debt items.

## Key Insights & Learnings

1. The reduced-motion guard must be per-method, not per-service — formalize as: "every method returning duration > 0 MUST have `if (this._reducedMotion) return 0`"
2. OCGCore locations/flags must be audited during story creation, not discovered during implementation
3. The code review AI remains the critical safety net — all reduced-motion omissions and data assumption errors caught before completion
4. Fire-and-forget + sync duration return is a proven, reusable animation integration pattern
5. CSS `@keyframes` that redefine `transform` will overwrite (not compose with) existing transforms on pseudo-elements — use separate keyframes or CSS variables
