## Successes & Strengths

1. **Protocol-first approach (Phase 0 Gate)** — Freezing `ws-protocol.ts` (49 message types) in Story 1.2 enabled all subsequent stories to work against a stable contract. Zero breaking changes across the entire epic.

2. **Story 1.1 skip decision** — FMA revealed fundamental incompatibility between solo grid (7-col, EMZ in grid, drag-drop) and PvP grid (6-col, EMZ external, click-based). Skipping saved significant wasted effort.

3. **Code review AI effectiveness** — ~79 findings across 6 stories, including 10+ Critical/High severity issues (race conditions, subscription leaks, network breaks, CSS bugs). All caught before merge.

4. **Previous Story Intelligence pattern** — Each story documents debug logs, variances, and lessons in Dev Notes for the next story. Created knowledge continuity across context windows.

5. **Tri-service architecture operational** — Angular ↔ WebSocket ↔ Duel Server ↔ HTTP ↔ Spring Boot fully connected by end of Epic 1.

6. **Architecture variances well-managed** — 5 variances detected, all justified:
   - `POST /api/duels/:id/join` removed → token-based association (1.3)
   - Node 24 instead of 22 (1.2)
   - 6 prompt sub-components instead of 3 (1.6 — UX spec precedence)
   - CrudRepository instead of JpaRepository (1.4)
   - HintContext fields adapted to actual protocol (1.4)

## Challenges & Growth Areas

1. **High volume of code review findings (~13 per story average)** — Indicates lack of pre-review checklist for common Angular pitfalls. The dev produces code that works but misses accessibility, touch targets, and CSS class binding patterns.

2. **`[class]` binding bug recurred in 3/6 stories** — The lesson from Story 1.5 was not applied in Stories 1.6 and 1.7. Angular's `[class]="..."` wipes base CSS classes; must use `[class.specific-class]` instead.

3. **Touch target non-compliance (3/6 stories)** — Interactive elements initially sized below 44-48px minimum. Fixed in review every time but should be caught at dev time.

4. **Story 1.7 too large** — Touched 5 existing components, created 3 new ones, orchestrated everything in DuelPageComponent. Exceeded context window (split across 2 windows). Auto mode activation toggle was completely stubbed — caught only in review.

5. **Integration wiring gaps** — Signals connected at type level but not actually bound in templates (zone browser `actionableCardCodes` in 1.7). Only detectable via thorough code review.

## Key Insights & Learnings

1. Code review is indispensable but the finding volume reveals a process gap — need dev-side checklist
2. Large stories (>3 new + 3 modified components) risk context window overflow and incomplete implementation
3. Inter-story lessons must be actively capitalized — recurring bugs prove passive knowledge transfer is insufficient
4. Architecture variances are healthy when documented — all 5 were justified improvements over the original plan
5. The "big bang" approach (no automated tests) places full quality burden on code review
